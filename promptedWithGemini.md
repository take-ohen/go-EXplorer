解説
.gitignore は、Gitのバージョン管理システムに対して、追跡（track）を意図的に無視させたいファイルやディレクトリを指定するためのファイルです。

/go-EXplorer: go build によって生成される実行ファイルです。これは開発者の各環境でソースコードから生成されるべきものであり、リポジトリに含めません。
*.test, *.out: go test コマンドで生成されるテスト用のバイナリやカバレッジプロファイルです。
/vendor/: Go Modules導入以前に使われていた依存ライブラリの管理ディレクトリです。現在は go.mod で管理するため、通常は無視します。
.vscode/, .idea/: VSCodeやGoLandなどのエディタが使用する設定ファイルです。個人の開発環境に依存するため、共有しません。
.DS_Store, Thumbs.db: macOSやWindowsが自動生成するシステムファイルです。プロジェクトとは無関係なため無視します。
.env: APIキーなどの機密情報や環境ごとの設定を記述するファイルです。セキュリティ上の理由からリポジトリに含めてはいけません。

package core

import (
	"os"
	"path/filepath"
	"sync"
)

// FileItem represents a file or directory found during scanning.
type FileItem struct {
	Path  string
	Name  string
	IsDir bool
	Size  int64
}

// Scanner provides high-speed directory scanning capabilities.
type Scanner struct {
	// TODO: Implement channels for async results
	// sem limits the number of concurrent goroutines to prevent resource exhaustion.
	sem chan struct{}
}

// NewScanner creates a new Scanner instance with a concurrency limit.
func NewScanner() *Scanner {
	return &Scanner{}
	// Limit concurrency to 32 to avoid "too many open files" errors.
	return &Scanner{
		sem: make(chan struct{}, 32),
	}
}

// Scan starts scanning the directory at root.
// It returns a channel that receives FileItem objects as they are found.
// The channel is closed when the scan is complete.
func (s *Scanner) Scan(root string) <-chan FileItem {
	out := make(chan FileItem, 1000) // Buffered channel for performance
	var wg sync.WaitGroup

	wg.Add(1)
	go func() {
		defer wg.Done()
		s.walk(root, out, &wg)
	}()

	// Monitor routine to close the channel when all workers are done.
	go func() {
		wg.Wait()
		close(out)
	}()

	return out
}

func (s *Scanner) walk(path string, out chan<- FileItem, wg *sync.WaitGroup) {
	// Acquire semaphore token
	s.sem <- struct{}{}
	defer func() {
		// Release semaphore token
		<-s.sem
	}()

	entries, err := os.ReadDir(path)
	if err != nil {
		// Permission denied or other errors are ignored in this prototype.
		return
	}

	for _, entry := range entries {
		fullPath := filepath.Join(path, entry.Name())

		// Get file info for size
		info, err := entry.Info()
		var size int64
		if err == nil {
			size = info.Size()
		}

		out <- FileItem{
			Path:  fullPath,
			Name:  entry.Name(),
			IsDir: entry.IsDir(),
			Size:  size,
		}

		if entry.IsDir() {
			wg.Add(1)
			go func(p string) {
				defer wg.Done()
				s.walk(p, out, wg)
			}(fullPath)
		}
	}
}

scanner.go の役割
このファイルは、指定されたディレクトリを高速にスキャンするためのコアロジックです。

目的: Goの並行処理機能（Goroutine）を使い、大量のファイルやディレクトリを効率的に探索する。
主要な構成要素:
FileItem 構造体: 見つかったファイルやディレクトリの情報を保持する。
Scanner 構造体: スキャン処理全体を管理する。
Scan() メソッド: スキャンを開始し、結果をリアルタイムで送り返すための「通り道」を返す。
walk() メソッド: 実際にディレクトリを再帰的に探索する内部関数。
<- (チャネル演算子) の解説
<- は、Goの並行処理で使われるチャネル (Channel) を操作するための演算子です。 チャネルは、Goroutine間で安全にデータを送受信するための「パイプ」や「通り道」のようなものです。

この演算子は、記述する位置によって2つの意味を持ちます。

1. channel <- data (送信)
意味: チャネルにデータを送信する。

scanner.go の walk 関数内に、この使い方の例があります。

go
// c:\Users\dora\home\src\go\go-EXplorer\internal\core\scanner.go
// ...
		out <- FileItem{
			Path:  fullPath,
			Name:  entry.Name(),
			IsDir: entry.IsDir(),
			Size:  size,
		}
// ...
構文分解:
out: chan<- FileItem 型のチャネル（データの通り道）。
<-: 送信演算子。
FileItem{...}: 送信するデータ。
図解:
text
+----------------+      +----------------+
| FileItemデータ |----->|  out チャネル  |
+----------------+      +----------------+
      (送信)
解説: walk 関数は、見つけたファイルの情報 (FileItem) を out という名前のチャネルに送信しています。これにより、スキャン処理の完了を待たずに、見つけたものから順次結果を外部に伝えることができます。
2. <- channel (受信)
意味: チャネルからデータを受信する。

scanner.go の walk 関数内に、この使い方の例があります。

go
// c:\Users\dora\home\src\go\go-EXplorer\internal\core\scanner.go
// ...
	// Acquire semaphore token
	s.sem <- struct{}{}
	defer func() {
		// Release semaphore token
		<-s.sem
	}()
// ...
構文分解:
<-: 受信演算子。
s.sem: chan struct{} 型のチャネル。
図解:
text
+----------------+      +----------------+
|  (空のデータ)  |<-----|  s.sem チャネル|
+----------------+      +----------------+
      (受信)
解説: <-s.sem は、s.sem チャネルからデータを受信しようとします。このコードでは、同時に実行できるGoroutineの数を制限するために使われています。
s.sem <- struct{}{}: s.sem チャネルにデータを送信し、実行権を1つ消費する。
<-s.sem: s.sem チャネルからデータを受信し、実行権を1つ返却する。
3. 型定義における <- (チャネルの方向)
型定義で使う場合、<- はチャネルが「送信専用」か「受信専用」かを示します。

<-chan FileItem: 受信専用チャネル。このチャネルからはデータを受け取ることしかできない。

// Scanメソッドは「受信専用」チャネルを返す
func (s *Scanner) Scan(root string) <-chan FileItem {
// ...
}

// walkメソッドは「送信専用」チャネルを引数に取る
func (s *Scanner) walk(path string, out chan<- FileItem, wg *sync.WaitGroup) {
// ...
}
解説: Scan メソッドを呼び出す側は、結果を受信するだけです。誤ってデータを送信してしまうことを防ぐために、型レベルで「受信専用 (<-chan)」と明記しています。 逆に walk 関数は、結果を送信するだけなので「送信専用 (chan<-)」と明記されています。 このように方向を限定することで、より安全なプログラムを書くことができます。

Go言語の並行処理（Goroutine）における「安全」とは、データ競合（Data Race）が発生しない状態（スレッドセーフ）を指します。

具体的に解説します。

1. 「安全ではない」状態（データ競合）
複数のGoroutineが、何の制御もなく同時に同じメモリ領域（変数）を読み書きすると、データが壊れたり、計算結果が矛盾したりします。これを「データ競合」と呼びます。

例：カウントアップ処理 変数 count = 0 に対して、2つのGoroutine（AとB）が同時に +1 しようとした場合：

Aが count (0) を読み込む。
Bが count (0) を読み込む。
Aが 0 + 1 = 1 を書き込む。
Bが 0 + 1 = 1 を書き込む。
結果: 本来 2 になるべき値が 1 になります。これが「安全ではない」状態です。

2. 「安全」な状態
上記の競合を防ぐために、一度に1つのGoroutineしかデータに触れないように制御されている状態です。

Go言語の chan（チャネル）は、この制御を自動的に行います。

排他制御: チャネルへの送信・受信操作は、Goのランタイム内部でロック（Lock）されています。
順序保証: あるGoroutineがデータを送信している最中に、別のGoroutineが割り込んでデータを壊すことはできません。必ず処理が完了してから次の処理が行われます。
まとめ
安全: 複数のGoroutineが同時に動いても、データの整合性が保たれ、プログラムが正しく動作すること。
手段: chan 型や sync パッケージ（Mutexなど）を使用することで、この「安全性」を確保します。

Defining for Clarity's Sake
「同期プリミティブ (Synchronization Primitive)」とは、並行して動く複数の処理（Goroutine）の足並みを揃えるために、プログラミング言語やOSが提供する最も基本的な仕組みのことです。

言葉を2つに分解して解説します。

1. 「同期 (Synchronization)」とは？
バラバラに動いている複数の処理のタイミングを合わせることです。

Goroutineは、起動するとそれぞれ勝手な速度で走り出します。放っておくと、以下のような問題が起きます。

まだ計算中なのに、結果を表示しようとしてしまう（順序の不整合）。
2つのGoroutineが同時に同じデータを書き換えて、データが壊れる（競合）。
これらを防ぐために、「ここで待て」「次は君の番」と制御することを「同期」と呼びます。

2. 「プリミティブ (Primitive)」とは？
**「原始的な」「基本的な構成要素」**という意味です。

複雑なシステムを作るための、これ以上分解できない最小単位の部品（道具）を指します。 プログラマは、これらの「基本的な部品（プリミティブ）」を組み合わせて、安全な並行処理プログラムを構築します。

Go言語における具体例
scanner.go のコードには、代表的な同期プリミティブが2つ登場しています。

A. sync.WaitGroup （完了の同期）
役割: 「全員が終わるまで待つ」
動作: バラバラに動くスキャン担当のGoroutineたちが、全て Done() するまで、メインの処理を Wait() で足止めします。これにより、「スキャン完了」というタイミングを同期させています。

動作: scanner.go の sem チャネル（セマフォとして使用）は、「バッファが空くまで送信を待つ」という動作をします。これにより、同時に走るGoroutineの数を32個に制限する（タイミングを制御する）同期を行っています。

Go言語の思想の核心と、それが scanner.go のコードにどのように反映されているかを、改めて解説させてください。

Go言語の思想の核心：「通信によるメモリの共有」
多くの言語では、複数の処理（スレッド）が同じデータ（メモリ）を直接触り、その際にデータが壊れないよう鍵（ロック）をかけて守ります。これを「メモリを共有することで通信する」と言います。これは非常に複雑で、鍵の管理を少しでも間違えるとプログラムは簡単に壊れます。

Go言語は、この考え方を逆転させました。

"Don't communicate by sharing memory; share memory by communicating." （メモリを共有して通信するな、通信によってメモリを共有せよ。）

これは、**「各処理は独立して動き、処理間でデータのやり取りが必要になったら、チャネルという安全な通信路を通して行う」**という思想です。各処理は他の処理が持つデータを直接触らず、通信に専念します。

scanner.go は、まさにこの思想の塊です
scanner.go の目的は「ファイルシステムを高速にスキャンする」ことです。ファイル操作は、CPUの計算速度に比べて非常に遅い「待ち時間」の多い処理です。Goの思想は、この「待ち時間」を有効活用するためにあります。

1. なぜGoroutine (go) を使うのか？
思想: 一つの巨大な仕事を、独立して動ける小さな仕事に分割する。
scanner.goでの実践:
「Cドライブ全体をスキャンする」という巨大な仕事を、walkという関数（仕事人）に任せます。
walkは、サブディレクトリを見つけるたびに、goキーワードを使って自分の分身（新しいwalkのGoroutine）を生成し、そのサブディレクトリの探索を任せます。
自分自身は、分身の完了を待たずに、同じ階層の別のディレクトリの探索を続けます。
これにより、無数のwalkたちが、ディスクのあちこちを**同時に（並行して）**探索し始めます。CPUがファイル読み込みを待っている間に、別のwalkが次の探索準備をする、というように無駄な時間がなくなります。
2. なぜチャネル (chan) を使うのか？
思想: 分割された仕事人たちが、安全に結果を報告し、互いに協調するための「通信路」を用意する。
scanner.goでの実践:
out チャネル（報告用の通信路）:
何千ものwalkたちが、見つけたファイル情報（FileItem）を、共有のリストに直接書き込む（危険な方法）のではなく、outという一方通行のチャネルに送信するだけです。
UI側（呼び出し元）は、このoutチャネルからデータが送られてくるのを待っているだけで、見つかった順にファイル情報を安全に受け取れます。これが「通信によるメモリの共有」です。
sem チャネル（調整用の通信路）:
あまりに多くのwalkたちが同時にファイルを開こうとすると、OSに「一度に開きすぎです」と怒られます（ファイルデスクリプタの枯渇）。
semチャネルは、**32個しか入れない「許可証の発行所」**として機能します。walkは仕事の開始前に許可証を1枚取り、終わったら返却します。許可証がなければ、誰かが返すまで待機します。
これにより、walkたちは互いの存在を直接知らなくても、「チャネルとの通信」を通じて全体の同時実行数が自然と調整されます。
3. なぜsync.WaitGroupを使うのか？
思想: 全ての仕事が終わったことを知るための、シンプルな合図。
scanner.goでの実践:
walkの分身がどれだけ生まれるかは、実行してみないと分かりません。
WaitGroupは「未完了の仕事の数」を数える単純なカウンタです。
新しいwalkを起動する前にwg.Add(1)でカウンタを増やし、walkの処理が終わる際にwg.Done()でカウンタを減らします。
監視役のGoroutineは、このカウンタが0になるのをwg.Wait()でひたすら待ちます。0になった瞬間、それは「全ての探索が終わった」という合図なので、報告用のoutチャネルをclose()して、大仕事の完了をUIに伝えます。
まとめ
scanner.goは、単に構文を並べたものではなく、Go言語の思想を体現した芸術品のようなものです。

仕事をGoroutineで細かく分割し、
チャネルという通信路で協調・報告させ、
WaitGroupで全体の完了をシンプルに検知する。
この「部品」の組み合わせこそが、Goが目指す「シンプルで、安全で、高性能な並行処理」の極意です。私の説明不足で、この最も重要な点をお伝えできなかったことを、重ねてお詫びいたします。
